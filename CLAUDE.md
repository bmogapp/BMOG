# CLAUDE.md

## Project
BMOG is a one-stop fitness courses, venue and event booking app (iOS / Android).
This is a monorepo: the Expo app lives at the repo root, the Java backend lives in `server/`.

## Stack
- Frontend: Expo SDK 56 (Expo Router v6) + TypeScript + NativeWind v4 + react-native-reusables
- Backend: Java 21 + Spring Boot 3.5.15 + Maven (`server/`) + Supabase Postgres (DB + Storage only)
- Payment: TapPay (前端 WebView 收卡 / 後端透過 Java REST 呼叫 TapPay Server API)
- OTP: Firebase Auth (前端拿 ID token / Java 後端用 Firebase Admin SDK 驗 token)
- Maps: react-native-maps (Google Maps)
- Email: Resend (透過 Java 後端 HTTP)
- Build: Expo EAS Build (iOS / Android)

## Commands

### Frontend (repo root)
- Start (dev client, not Expo Go): `npx expo start`
- iOS simulator (build + launch dev client via Xcode): `npx expo run:ios`
- Android emulator: `npx expo run:android`
- Build (EAS): `eas build --platform all`
- Test single: `npx jest src/__tests__/[file]`
- Test all: `npx jest`
- Lint: `npx eslint . --fix`
- Type check: `npx tsc --noEmit`

> ⚠️ This project uses `expo-dev-client`, not Expo Go. Native modules (react-native-maps,
> Firebase Auth, TapPay) aren't supported in Expo Go, so always run via `npx expo run:ios` /
> `run:android` (or a custom EAS dev-client build) — never scan the QR code with the Expo Go app.

### Backend (`server/`)
- Run locally: `cd server && ./mvnw spring-boot:run`
- Build: `cd server && ./mvnw clean package`
- Test: `cd server && ./mvnw test`

## Architecture

### Frontend (repo root)
- src/app/              → screens and routes (file-based routing)
- src/app/(auth)/       → login, register, OTP screens
- src/app/(tabs)/       → main tab navigator
- src/components/       → stateless UI components (NativeWind styled)
- src/lib/services/     → Supabase queries and business logic
- src/hooks/            → custom React hooks
- src/lib/utils/        → shared helpers
- src/lib/types/        → TypeScript types
- supabase/functions/   → Edge Functions
- supabase/migrations/  → DB migrations (auto-managed)

### Backend (`server/`, groupId `com.bmog`, package `com.bmog.server`)
- server/src/main/java/com/bmog/server/        → application code
- server/src/main/resources/application.properties → config (never put secrets directly here, use env vars)
- server/src/test/java/com/bmog/server/        → tests
- Handles: Firebase ID token verification (Firebase Admin SDK), TapPay Server API calls, Resend email HTTP calls
- Does NOT handle: DB writes/migrations (Supabase Postgres) or webhook receipt (Supabase Edge Functions) — those stay on the Supabase side

## Rules
- NEVER commit .env files, secrets, or google-services.json
- All Supabase queries through src/lib/services/, never in components or screens
- All async calls must use try/catch
- TapPay WebView logic must stay isolated in its own component
- Firebase Auth only used for OTP flow; all other auth via Supabase Auth
- Prefix commits: feat:, fix:, docs:, refactor:, test:, chore:
- IMPORTANT: run type check after every code change
- No console.log in production code (use a logger utility if needed)

## Workflow
- Ask before making architectural decisions
- Make minimal changes, don't refactor unrelated code
- Run tests after every change, fix failures before moving on
- Create separate commits per logical change
- When unsure between two approaches, explain both and let me choose

## Out of scope
- supabase/migrations/  → managed by Supabase CLI, don't create manually
- assets/               → static files, don't modify
- .expo/                → auto-generated cache, don't touch
- android/ / ios/       → generated by EAS or prebuild, don't edit directly without asking
- server/target/        → Maven build output, don't edit
